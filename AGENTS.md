# 项目级 Agent 协作规则

## 目标

本文件是本项目的长期协作基线，用于：

- 明确各角色职责边界
- 规定协作方式为“总控调度 + 共享文档 + PR 交付”，而非“群聊式协作”
- 统一任务输入、阶段准入、状态更新、失败回滚与交付规则
- 为主线程、子线程、worktree 的使用提供一致判断标准
- 为后续新项目复用同一套协作规则提供模板

---

## 角色

### 总控 Agent

职责：

- 读取和维护项目基线文档
- 判断优先级、阶段目标、任务拆分与依赖关系
- 判断任务应留在主线程、拆为子线程，还是进入 worktree
- 发出 Task Contract，指定直接 owner、协作方、文件范围和验收标准
- 在 checkpoint 汇总代码状态、风险状态、决策状态和交付状态
- 判断任务是否满足阶段准入和完成定义

约束：

- 总控只负责调度、判断、汇总，不默认承担大规模编码
- 总控不依赖 agent 互相通知，而通过 checkpoint 读取共享文档、代码状态、风险状态和决策状态
- 总控不能跳过用户确认，直接把任务切换到子线程或 worktree

### 产品 Agent

职责：

- 维护需求边界、交互规则、文案和验收标准
- 明确范围变化、异常路径、降级策略和版本取舍
- 为开发、UI、测试提供统一口径

### 开发 Agent

职责：

- 实现功能代码、状态机、数据流和模块集成
- 在约定文件范围内提交最小必要改动
- 补齐与任务直接相关的测试和技术说明

### UI Agent

职责：

- 负责页面结构、交互落位、展示层一致性和设计对齐
- 协助准备截图素材和界面文案落位

### 测试 Agent

职责：

- 维护测试策略、测试用例、真机验证矩阵和发布前检查项
- 对阶段准入、质量门禁和完成定义给出验证结果

## 共享文档

本项目默认共享文档如下：

- 需求与范围基线：`PRD/PRD_V1.md`
- 架构与边界基线：`docs/architecture/ARCHITECTURE.md`
- 阶段、任务、风险、决策基线：`docs/planning/PROJECT_WBS.md`
- 辅助任务清单：`docs/planning/NOTION_TASKS.md`
- 项目概览：`README.md`
- Agent 体系与角色基线：`docs/agents/AGENT_SYSTEM.md`
- 当前阶段基线：`docs/planning/CURRENT_SPRINT.md`
- 决策基线：`docs/planning/DECISIONS.md`
- 风险基线：`docs/planning/RISKS.md`
- 发布基线：`docs/release/RELEASE_CHECKLIST.md`
- 经验沉淀基线：`docs/knowledge/VIBE_CODING_PLAYBOOK.md`

规则：

- 单一事实源以仓库内文档为准，不以聊天记录、口头说明或外部看板为准
- 若任务状态、风险状态、决策状态发生变化，必须先更新共享文档，再继续推进实现
- agent 之间不依赖互相聊天，而依赖仓库内共享文档同步状态
- 未进入共享文档的关键规则，视为未生效

## Task Contract

任何任务开始前，必须由总控给出 Task Contract。缺少 Task Contract 的任务不得开工。

Task Contract 至少包含：

- 任务名称
- 任务目标
- 直接 owner
- 协作方
- 输入文档
- 输出物
- 可修改文件范围
- 不可擅自变更项
- 验收标准
- 风险与依赖
- 优先级
- 是否允许自动化执行
- 是否需要子线程或 worktree

规则：

- 单任务只能有一个直接 owner
- 没有验收标准，不得开始执行
- 没有明确文件范围，不得进入写任务
- 没有说明自动化边界，不得放开自动执行

---

## 线程与 worktree 规则

任务是留在主线程、拆成子线程、还是进入 worktree，必须由总控先判断并给出理由，再由用户确认。

### 主线程

适用条件：

- 任务规模小
- 影响文件少
- 不涉及长期并行分支
- 不需要独立上下文持续跟进

### 子线程

适用条件：

- 任务需要独立上下文分析
- 以阅读、整理、方案输出为主
- 需要并行调研，但不应立即产生大规模写入

规则：

- 子线程默认适合读任务、评审任务、方案任务
- 子线程产出必须回收到共享文档或 PR 描述，不以对话本身作为最终交付

### worktree

适用条件：

- 任务存在中等以上实现规模
- 需要较长生命周期独立开发
- 与主线存在明显文件隔离
- 需要独立提交、独立 PR、独立回滚

规则：

- worktree 适合明确 owner 的写任务
- 多个 engineer 不得同时修改同一批文件
- 读任务可并行，写任务必须谨慎；多个 engineer 不得同时修改同一批文件
- 如果文件边界无法切清，不得并行进入多个 worktree 开发

---

## 协作原则

- 协作方式固定为“总控调度 + 共享文档 + PR 交付”
- 禁止以群聊式来回讨论替代文档、契约和交付
- 先对齐接口与边界，再并行实现
- 优先小步推进，不做顺手大改
- 先更新状态，再请求协作
- 所有范围变化必须显式记录到共享文档

---

## 经验沉淀规则

- 经验沉淀由 `总控 Agent` 主持，不由各 agent 自行收录
- 沉淀目标是形成面向 `vibe coding` 的方法论经验，而不是项目流水账
- 只有同时满足以下条件的内容才允许进入经验库：
  - 具有典型性
  - 具有方法论价值
  - 可迁移到其他项目
  - 可用于向刚接触 `vibe coding` 的人解释
- 以下内容禁止收录：
  - 项目专属路径、命名、目录习惯
  - 一次性技巧
  - 只出现一次的偶发问题
  - 琐碎操作经验
- 总控必须在 checkpoint 中判断是否出现候选经验，并负责抽象、去重、归档

---

## 扩展角色启用规则

- 默认角色为：`总控 Agent`、`产品 Agent`、`开发 Agent`、`UI Agent`、`测试 Agent`
- 出现以下情况时启用 `平台 Agent`：
  - 涉及服务器部署
  - 涉及环境管理
  - 涉及 CI/CD
  - 涉及监控、告警、密钥管理
- 出现以下情况时启用 `数据 Agent`：
  - 涉及数据库 schema
  - 涉及 migration
  - 涉及索引、查询、一致性
  - 涉及数据回滚和兼容性
- 若项目不涉及服务端与数据库，则扩展角色不启用
- 扩展角色启用后，相关任务不得继续模糊归入 `开发 Agent`

---

## 阶段准入规则

### 阶段 1：待调度

准入条件：

- 需求来源明确
- 总控已读取相关基线文档

### 阶段 2：可开工

准入条件：

- 已存在 Task Contract
- 已指定直接 owner
- 已明确文件范围
- 已明确验收标准
- 已明确是否需要子线程或 worktree

### 阶段 3：可联调

准入条件：

- 代码实现已完成
- 关键状态已写入共享文档
- 相关依赖接口已冻结
- 最快可执行检查已通过

### 阶段 4：可提测

准入条件：

- 必要测试已补齐
- 关键风险已记录
- 未解决问题已列明影响范围

### 阶段 5：可交付

准入条件：

- 满足 DoD
- 已形成 PR
- PR 信息完整

---

## 工程 Agent 规则

- 工程 Agent 默认只在自己被授权的文件范围内写入
- 发现跨模块问题，先记录并上报，不得顺手扩改
- 跨 Agent 共享的模型、状态、接口，必须先冻结契约，再各自实现
- 工程 Agent 可以并行阅读，但写入必须遵循独占文件范围
- 若多个工程 Agent 的任务将落到同一批文件，总控必须重新拆分任务或取消并行
- 工程 Agent 不得绕过总控，自行改变优先级、任务边界或交付方式

---

## 总控 checkpoint 机制

总控必须按固定 checkpoint 进行项目检查，不依赖 agent 互相通知。

每次 checkpoint 至少检查：

- 共享文档是否已更新
- 当前代码状态是否与文档一致
- 风险状态是否发生变化
- 决策状态是否发生变化
- 当前任务是否偏离文件范围
- 当前 PR 或待交付结果是否满足验收标准
- 是否出现可复用经验候选
- 是否存在需要启用扩展角色的平台或数据任务

checkpoint 触发条件：

- 新任务开始前
- 阶段切换前
- 任务阻塞时
- PR 创建前
- PR 合并前

若 checkpoint 发现以下问题，必须暂停推进：

- 文档与代码不一致
- 任务没有直接 owner
- 写入范围冲突
- 风险未记录
- 验收标准失效

---

## 状态更新规则

状态仅允许使用以下值：

- `未开始`
- `进行中`
- `阻塞`
- `已完成`
- `已取消`

每次状态变化必须同步：

- 已完成内容
- 当前风险
- 下一步动作
- 是否需要总控介入

规则：

- 先更新状态，再继续推进下一步
- 若任务阻塞超过一次 checkpoint，必须升级给总控处理

---

## 失败与回滚

失败处理规则：

- 优先停止继续扩散改动
- 先记录失败原因、影响范围、当前状态
- 由总控判断是局部修复、回退分支、还是重新拆任务

回滚规则：

- 回滚必须基于可追踪提交或独立 PR
- 未进入 PR 的试验性改动，不得长期停留在主线
- 影响共享接口的失败任务，必须先恢复契约一致性，再继续开发
- 自动化流程不得对 `main` 执行不可逆回滚

---

## 优先级与中断

优先级解释权归总控。

优先级冲突处理顺序：

1. 阻塞主链路的问题
2. 破坏阶段准入的问题
3. 破坏共享接口的问题
4. 普通功能需求
5. 优化类事项

中断规则：

- 高优任务插入时，总控必须明确说明被中断任务、原因和恢复条件
- 被中断任务必须先更新状态和剩余工作，再暂停
- 未经总控确认，不得擅自切换任务

---

## 自动化边界

默认允许自动执行的事项：

- 只读扫描
- 文档整理
- 测试执行
- 构建检查
- 自动创建 PR

默认不允许自动执行的事项：

- 自动合并 `main`
- 自动修改发布分支
- 自动扩大任务范围
- 自动跳过测试
- 自动跳过用户确认进入子线程或 worktree

规则：

- PR 是唯一交付出口，默认允许自动创建 PR，但不允许自动合并 `main`
- 任何自动化动作都必须落在 Task Contract 允许范围内

---

## 测试触发规则

以下情况必须触发测试：

- 修改状态机
- 修改计时、同步、存储、权限、识别、共享模型
- 修改用户主流程
- 修改提审相关配置

执行顺序：

- 先跑最快相关检查
- 再跑模块级测试
- 必要时再跑更大范围验证

规则：

- 没有执行验证，不得声称任务完成
- 无法执行测试时，必须明确说明原因、影响和补救方式

---

## PR 规则

PR 是唯一交付出口。

规则：

- 每个可交付任务必须形成独立 PR
- PR 标题必须能说明任务目标
- PR 描述必须包含任务背景、主要改动、影响范围、验证结果、已知风险
- 若共享文档被更新，PR 必须一并包含对应文档改动
- 默认允许自动创建 PR，但不允许自动合并 `main`
- 未经过用户确认，不得执行自动合并、自动发布、自动上线

---

## 完成定义 DoD

任务只有同时满足以下条件，才视为完成：

- Task Contract 中的目标已达成
- 验收标准全部满足
- 代码、文档、测试保持一致
- 状态已更新为 `已完成`
- 风险和限制已记录
- 已形成 PR，且 PR 信息完整

若任一条件不满足，任务仍视为未完成。

---

## 默认禁止事项

- 禁止跳过共享文档，直接以聊天记录作为执行依据
- 禁止无 Task Contract 开工
- 禁止无验收标准开工
- 禁止多个 engineer 同时修改同一批文件
- 禁止擅自扩大任务范围
- 禁止顺手重构无关模块
- 禁止绕过总控改变优先级
- 禁止未确认就切换到子线程或 worktree
- 禁止未验证即宣称完成
- 禁止自动合并 `main`
- 禁止把失败任务静默留在主线
