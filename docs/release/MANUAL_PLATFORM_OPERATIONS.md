# 平台 / 设备手动操作 Runbook

## 文档目标

本文件用于明确：

- 哪些平台 / 设备操作必须由用户亲自执行
- 为什么 Agent 不能代做
- 用户执行后应如何回传结果
- 当前项目中与真实设备、系统权限、签名、平台账号相关的统一手工验证入口

本文件是 `RELEASE_CHECKLIST.md`、`TEST_MATRIX.md` 的补充说明：

- `RELEASE_CHECKLIST.md` 负责“要检查什么”
- `TEST_MATRIX.md` 负责“哪些场景必须覆盖”
- 本文负责“用户要怎么实际操作，以及如何把结果回传给总控”

---

## 适用范围

以下动作默认属于“必须由用户手动完成”的平台 / 设备操作：

- 真实 `iPhone + Apple Watch` 配对、解锁、佩戴与现场观察
- 系统权限弹窗点击与授权路径选择
- `Health` / `Health app` 中的实际写入结果确认
- Apple Developer / 签名 / Provisioning / capability 的账号侧操作
- Xcode 中依赖真实设备、账号或本机 GUI 状态的操作
- App Store Connect / TestFlight / 提审后台类操作

若某项操作满足以下任一条件，也应视为手动平台操作：

- 需要真实设备上的物理交互
- 需要系统弹窗点击
- 需要登录个人或组织平台账号
- 需要观察 Agent 无法直接感知的设备 UI / 体感 / 系统结果

---

## 为什么 Agent 不能代做

Agent 不能可靠代做这类动作，原因包括：

1. 无法替代真实用户在设备上的物理操作。
2. 无法安全接管用户的 Apple ID、开发者账号、证书与签名身份。
3. 无法可信地观察手表佩戴、系统授权弹窗、Health app 实际写入结果等 GUI / 设备态信息。
4. 某些结果天然依赖真实配对状态、系统设置、传感器和本机环境，Agent 即使能改代码，也不能伪造真实验证结论。

因此，Agent 只负责：

- 冻结边界
- 准备 runbook
- 明确回传格式
- 基于用户回传结果更新共享文档和做 checkpoint 判断

---

## 通用回传格式

用户完成任一手动平台操作后，统一按以下格式回传给总控线程：

```md
## 手动操作回传
- 操作名称：
- 执行日期：
- 执行设备：
- 执行环境：
- 实际步骤结果：
- 观察到的现象：
- 异常 / 偏差：
- 结论：
```

填写要求：

- `执行设备`：至少写清 iPhone 型号、Apple Watch 型号、系统版本
- `执行环境`：至少写清是否真实配对、是否首次安装、是否首次授权
- `实际步骤结果`：按 runbook 中的步骤逐条写“通过 / 失败 / 不适用”
- `异常 / 偏差`：若没有，明确写 `无`
- `结论`：只能写本次操作实际验证到的结论，不要扩大表述

## TASK_016 真实动作计数最小真机验证

适用前提：

- 当前分支为 `codex/task-016-real-live-detection`
- 当前目标是验证第一阶段“真实可用闭环”，不是宣称高准确率稳定
- 使用真实配对的 `iPhone + Apple Watch`
- 使用 Xcode 将当前构建安装并运行到真实 Apple Watch
- Apple Watch 已佩戴、已解锁、可正常启动 App

本轮必须由用户手动完成的最小动作：

1. 首次无校准结果时，点击开始训练，确认进入一次性校准。
2. 按页面提示完成一次校准，确认校准完成后进入训练倒计时。
3. 完成一组正常深蹲，观察 `live` 计数是否随真实动作增加。
4. 重新开始训练或复用已有校准，确认不再强制重复校准。
5. 在倒计时阶段取消，确认未产生新计数。
6. 在训练中暂停，再做轻微动作，确认暂停期间不继续累计新计数。
7. 在休息或完成状态下做轻微动作，确认不继续累计新计数。
8. 观察正式用户路径是否仍出现“模拟识别”或 `+1 / -1`；若出现，记录具体页面与状态。
9. 选做但建议执行：半蹲、抬腕、轻微晃动各 1-2 次，记录是否误计。

最小回传模板：

```md
## TASK_016 真实动作计数手动回传
- 执行日期：
- 执行设备：
- 执行环境：
- 当前分支 / 构建：
- 首次校准：通过 / 失败；观察：
- 校准复用：通过 / 失败；观察：
- 正常深蹲计数：通过 / 失败；观察：
- 倒计时取消不计数：通过 / 失败；观察：
- 暂停期间不计数：通过 / 失败；观察：
- 休息 / 完成状态不计数：通过 / 失败；观察：
- 正式路径无模拟识别与 +1/-1：通过 / 失败；观察：
- 半蹲 / 抬腕 / 晃动防误触：通过 / 失败 / 未执行；观察：
- 异常 / 偏差：
- 结论：
```

判定口径：

- 若首次校准、校准复用、正常深蹲计数、非训练态不计数任一失败，不得宣称“真实可用闭环已通过”。
- 半蹲、抬腕、轻微晃动若仍有误计，应记录为调优项；是否阻塞本轮由总控按影响范围判断。
- 单次真机通过只能说明“本次最小闭环通过”，不得扩大为“高准确率稳定”。
- 若“通过 / 失败”字段与原始观察冲突，以原始观察为准。

## 6.9 免费 Apple 个人开发者最小手动清单（Personal Team）

适用前提：

- 当前账号为免费 `Apple Personal Team`
- 当前目标是 `6.9` checkpoint，不是提审/分发

本轮不要求执行的平台动作：

- `App Store Connect` 的 `App Privacy` 后台问卷
- `TestFlight` 上传与分发
- 提审材料上传、审核问答与分发型签名/发布动作

本轮必须手动完成的最小动作：

1. 本地 Xcode 安装并运行到真实配对 `iPhone + Apple Watch`
2. 完成训练后确认 `Health app` 新增 workout
3. 倒计时取消后确认 `Health app` 未新增 workout

最小回传判定规则：

- 若“结果（通过/失败）”与“观察到的现象”冲突，以原始观察为准
- 出现冲突时，在 checkpoint 里标注“字段误填，按原始观察归一”
- 倒计时取消场景建议同时回传取消时刻与 `Health app` 记录时刻，避免与历史记录混淆

## 6.8 手动验证前提

执行 `6.8 HealthKit 与权限` 相关手动操作前，用户需先确认：

- 使用真实配对的 `iPhone + Apple Watch`
- 使用可运行当前构建的本机与 Xcode 环境
- Apple Watch 已佩戴、已解锁、可正常启动 App
- 当前构建已安装到真实手表
- 若要验证首次授权，需要确保当前 App 的 `HealthKit` 权限状态可回到“未决定”或使用全新安装路径

若上述条件不成立，不应宣称 `6.8` 真机验证已完成。

## 6.8-1 HealthKit 首次授权

### 目标

确认首次进入需要 `HealthKit` 的训练路径时：

- 系统会正确触发首次授权
- 授权文案与真实用途一致
- 授权成功后不会阻塞 Watch 训练主流程

### 用户手动步骤

1. 在真实配对的 `iPhone + Apple Watch` 上安装当前构建。
2. 确保当前 App 对应的 `HealthKit` 状态为首次授权场景。
3. 启动 Apple Watch App，并进入会触发 `HealthKit` 的训练路径。
4. 观察是否出现系统授权弹窗。
5. 选择“允许”或等价授权路径。
6. 继续完成一次最小训练流程，确认主流程未因授权而中断。
7. 记录授权后页面状态、训练是否正常开始、是否出现异常提示。

### 用户回传重点

- 是否出现首次授权弹窗
- 授权文案是否符合“训练运行 / 写入 Health app workout”的真实用途
- 授权后是否能继续进入训练主流程
- 是否出现崩溃、白屏、卡死、重复弹窗

---

## 6.8-2 HealthKit 拒权降级

### 目标

确认用户拒绝 `HealthKit` 后：

- Watch 本地训练主流程仍可继续
- App 不会因为拒权崩溃或锁死
- 产品口径符合“训练可继续，但不会写入 Health app”

### 用户手动步骤

1. 在真实设备上进入 `HealthKit` 首次授权路径。
2. 在系统授权弹窗中选择“拒绝”或等价拒权路径。
3. 返回 App，继续尝试开始一次最小训练流程。
4. 完成倒计时、训练、暂停 / 恢复或结束等最小主流程观察。
5. 确认 App 是否仍可继续训练。
6. 训练结束后检查是否错误声称已写入 `Health app`。
7. 记录拒权后页面提示、可继续执行的能力、被禁用的能力。

### 用户回传重点

- 拒权后是否还能开始与完成训练
- 是否有清晰的非阻塞提示
- 是否误写入 `Health app`
- 是否出现崩溃、空白页、权限页循环、训练硬阻塞

---

## 6.8-3 真实 Apple Watch 上 workout 写入 Health app

### 目标

确认一次已完成的真实训练在满足授权条件时：

- 可以写入 `Health app`
- 写入时机与冻结边界一致
- 写入结果与训练最小摘要一致

### 用户手动步骤

1. 在真实配对设备上，确保 `HealthKit` 已授权。
2. 在 Apple Watch 上启动 App 并开始一次最小训练。
3. 完成一轮可判定为“训练完成”的真实或最小模拟训练。
4. 结束后在 `Health app` 或对应系统入口中检查是否新增 workout 记录。
5. 观察记录的生成时机是否发生在训练完成之后。
6. 对照训练时长、组数、次数等最小摘要，检查是否存在明显错写或重复写入。
7. 另外补做两条负向观察：
   - 倒计时取消后是否未产生 workout
   - 中途结束训练后是否符合当前冻结口径（默认不写入）

### 用户回传重点

- `Health app` 中是否出现新的 workout
- 写入是否只发生在训练完成路径
- 是否存在重复写入
- 倒计时取消 / 中途结束是否误写入
- 训练摘要与写入结果是否明显不一致

---

## 6.8 手动验证结论口径

用户回传结论时，必须使用以下口径约束：

- 可以说：`HealthKit 首次授权通过`
- 可以说：`拒权降级符合预期`
- 可以说：`真实 Apple Watch 上 workout 写入通过`
- 不可以说：`6.8 已完全稳定`
- 不可以说：`HealthKit 所有平台问题已解决`

若只完成了单次真机验证，结论只能限定在“本次验证通过 / 失败”，不能扩大成全面稳定结论。

---

## 总控使用方式

总控在读取用户回传后，应至少判断：

1. 是否覆盖了首次授权、拒权降级、真实写入三类场景。
2. 是否写清执行设备与环境。
3. 是否存在与 `TASK_011` 冻结边界冲突的观察结果。
4. 是否需要把结果同步回 `CURRENT_SPRINT.md`、`PROJECT_WBS.md`、`RISKS.md`。
5. 是否满足进入下一 checkpoint / PR / 合入的准入条件。

若用户未按本文回传，默认视为“真机平台验证信息不足”。
