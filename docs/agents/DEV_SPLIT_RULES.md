# 开发 Agent 拆分规则

## 触发条件

满足任一条件时，可以将 `开发 Agent` 拆为 `开发 Agent 1 / 开发 Agent 2`：

- 存在两条以上可长期并行的实现主线
- 单个开发 Agent 的上下文明显过载
- 两组任务长期修改不同文件范围
- 已出现排队、等待或互相阻塞
- 需要独立 PR 和独立回滚能力

## 拆分原则

### 原则 1：按稳定边界拆

优先按以下边界拆分：

- 主流程 / 状态机
- 数据 / 同步 / 存储
- 算法 / 识别 / 规则
- 表现层 / 组件层

### 原则 2：按文件范围落地

拆分时必须同时明确：

- 各自可修改目录
- 各自主要 owner 文件
- 共享只读依赖
- 禁止主动修改的区域

### 原则 3：共享契约先冻结

若两个开发 Agent 共享模型、状态、协议、事件：

- 先冻结契约
- 再并行实现
- 契约变化必须先回写文档

### 原则 4：单任务仍然单 owner

- 一个任务不能由两个开发 Agent 共同 owner
- 若存在共同影响，由总控重新拆任务

## 建议拆分方式

### 方式 A

- `开发 Agent 1`：主流程 / 状态机 / 关键页面
- `开发 Agent 2`：同步 / 存储 / 集成

### 方式 B

- `开发 Agent 1`：主流程 / 业务实现
- `开发 Agent 2`：算法 / 识别 / 规则引擎

### 方式 C

- `开发 Agent 1`：核心功能
- `开发 Agent 2`：支线功能和工程化补齐

## 优先升级为扩展角色的情况

- 若任务核心问题是部署、环境、CI/CD、密钥、监控，不优先继续拆 `开发 Agent`，而应启用 `平台 Agent`
- 若任务核心问题是 schema、migration、一致性、回滚，不优先继续拆 `开发 Agent`，而应启用 `数据 Agent`
- `开发 Agent` 负责实现，不应长期兼任平台与数据 owner

## 禁止事项

- 禁止在文件边界不清时强行拆分
- 禁止两个开发 Agent 同时修改同一批文件
- 禁止以聊天约定代替书面契约
